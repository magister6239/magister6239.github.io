Software architecture

Last updated on Jan 1, 0001 5 min read
Image credit: John Moeses Bauan

What is software architecture for?

Software architecture is the foundation of IT product development. Developers write code to create a program. Strings are combined into functions, functions are combined into classes, classes are combined into modules. The result is a finished program.

To reduce the time required to create a complex product, developers at the beginning of work create a general plan for the future software with the main blocks and ways to connect them. Then they detail it, working out the requirements and characteristics. This is how information is obtained to build software components.

First, due to manageability, the architect uses individual components and modules responsible for individual functions to organize software development and support.

Also, the clear architecture of the software product makes it easy to support. Even if developers are not familiar with the details of the system, they will be able to quickly and easily navigate the code, making the team’s work easier.

With the help of software architecture, you can easily modify, expand and scale the project. Quickly add new elements, increase functionality and productivity, and use resources efficiently.

Software architecture, including monolithic architecture, allows for reuse. Components are used multiple times in different parts of the system. The main benefit is to reduce the number of errors and reduce development time. It also includes safety measures to ensure protection.

Basic principles of software architecture

Principles that are taken into account during design.

  • Flexibility. Build not for years, but with the possibility of making changes. Consider how the program may change to meet modern needs, and add flexibility to easily support changes.
  • Openness/closedness. Software entities are closed for modification and open for expansion.
  • Risk reduction and analysis model. It is important to use tools such as UML to capture requirements and design decisions. You should not formalize the model in such a way that it is impossible to easily adapt and repeat the design.
  • Stability - organizing dependencies from more stable components to less stable ones.
  • Uniqueness - avoiding code duplication.
  • Minimal knowledge. Each module knows only about its neighbors to reduce interdependence and increase isolation.
  • Sharing of responsibilities. 1 element - 1 reason for change (for a clear division of responsibilities).

By following these architectural principles, you can design a system that is extensible, flexible, and maintainable.

Types of architectural styles

Architectural styles are high-level strategies. They represent an abstract structure for systems.

8 main styles:

  • Multi-layered.
  • Component-oriented.
  • Service-oriented.
  • Distributed system.
  • Subject-oriented.
  • “Translator”.
  • Parallelism.
  • Data-driven.

Multilayer is the most common. A typical example is a layered style with 3 layers: presentation layer, business logic and data storage. Pros: easy to understand, test and support. The downside is the appearance of performance overhead.

Component-oriented separates functionality into layers within a single system, organized as reusable and loosely coupled components. Advantages: Flexibility, Maintainability, Reusability. Disadvantage: the components are difficult to manage.

Service-oriented for developing software as a set of interacting services over a network. Pros: Loose coupling, flexibility, and reusability. Disadvantages: complexity, network dependence.

Distributed system. Components on networked computers pass messages to communicate and coordinate actions to achieve a goal. Advantages: general access, resistance to errors and failures, scalability. Disadvantages: complexity, network dependence.

Domain-oriented - creation of software abstractions (domain models). It is characterized by limited connections, integrity and interconnection. Disadvantage: Requires deep understanding and experience.

The ‚Äútranslator‚Äù determines how sentences are evaluated. For each symbol, a terminal or non-terminal style is created. It is highly flexible and easy to debug. Disadvantage: Requires more resources.

In the parallelism style, multiple independent tasks are executed at the same time. Parallelism improves performance, which is especially important for multiprocessor systems. The downside is the complex mutual blocking.

Data-centric architecture. The style focuses on organizing and transforming data. Suitable for large volumes of data and solving complex problems. Ensures correct data sequencing, integrity and reliability. Disadvantages include possible scalability issues.

Architectural Patterns

These are subject-specific and specific elements for a particular module or problem. A structured solution that describes the organization of components and interactions. Patterns are similar to software design patterns, but more abstract.

Each architectural style has its own patterns. Multilayer has a multi-layer pattern and ‚ÄúClean Architecture‚Äù. The first divides the system into a certain number of layers, each with its own area of responsibility. The ‚ÄúClean Architecture‚Äù pattern emphasizes the separation of functionality in the system, organizing software into concentric layers, in the central one - the subject area, in a circle - specific layers for the application, the internal ones influence the external ones. So the changes have almost no effect on the business logic.

There are 3 patterns in the component-oriented style. Object-oriented pattern is a paradigm with objects containing data in the form of fields (attributes) and code in the form of procedures (methods). Facilitates the design of complex systems, their implementation and support. “Microkernel” to separate the microkernel from the extended client-specific functionality. This way you can easily and quickly expand the system without changing the main code. Plugin pattern - adds new functionality using modules or plugins, integrating them through a standard interface to expand and customize the application.

Service-oriented style patterns:

  • SOA. A software project is a set of individual services that are used in systems.
  • Broker. Coordinates the communication of system components through an intermediary.
  • Microservice pattern for architecture. An application project looks like a collection of small services. Each service communicates through lightweight mechanisms and operates in its own process.
  • Serverless pattern. Applications are created and run without servers in cloud environments.

The distributed system style has 2 patterns:

  • Data space. Evenly distributes resources and services across servers, which avoids a single point of failure.
  • Peer-to-peer network. Here, each participant acts as a client. Nodes and service exchange data without a central server.

By understanding styles and patterns, a specialist will be able to design reliable, scalable and maintainable systems.
